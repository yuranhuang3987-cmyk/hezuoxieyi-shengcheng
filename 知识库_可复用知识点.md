## 1. WSL2 代理配置标准流程

### 适用场景
WSL2 需要使用 Windows 代理上网

### 标准步骤

```bash
# 1. 获取 Windows IP（动态）
WINDOWS_IP=$(cat /etc/resolv.conf | grep nameserver | awk '{print $2}')

# 2. 设置代理环境变量
export http_proxy="http://${WINDOWS_IP}:21882"
export https_proxy="http://${WINDOWS_IP}:21882"
export ALL_PROXY="socks5://${WINDOWS_IP}:21881"

# 3. 写入 ~/.bashrc 永久生效
# 4. 测试：curl https://api.ipify.org
```

### 关键要点
- Windows 代理必须允许局域网连接（0.0.0.0，不是 127.0.0.1）
- Windows 防火墙要开放端口
- IP 每次启动 WSL2 可能变化，所以要动态获取

### 常见问题

**Q: 端口连不上？**
```
A: 检查 Windows 代理是否开启"允许局域网连接"
```

**Q: 代理无效？**
```
A: 确认环境变量已设置（echo $http_proxy）
```

**Q: 重启后失效？**
```
A: 已写入 ~/.bashrc，重新打开终端或 source ~/.bashrc
```

## 2. 搜索工具选择策略

### 决策树

```
需要搜索？
├─ 有 Brave API Key？
│  ├─ Yes → 使用 web_search（最快）
│  └─ No → 继续判断
├─ 需要精确页面内容？
│  ├─ Yes → 使用 web_fetch（最稳定）
│  └─ No → 使用 ddgs（可能超时）
└─ 搜索内容语言？
   ├─ 英文 → ddgs 成功率高
   └─ 中文 → ddgs 可能超时，优先 web_fetch
```

### 工具对比

| 工具 | 速度 | 稳定性 | 中文支持 | API Key |
|------|------|--------|----------|---------|
| web_search | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ✅ | 需要 |
| web_fetch | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ✅ | 不需要 |
| ddgs | ⭐⭐⭐ | ⭐⭐ | ⚠️ | 不需要 |

### 最佳实践

1. **优先级**：web_fetch > web_search > ddgs
2. **语言选择**：搜索用英文，比中文稳定
3. **失败处理**：ddgs 失败就换关键词重试
4. **代理配置**：ddgs 需要代理环境变量

### 技巧

**快速抓取网页内容：**
```bash
web_fetch(url, maxChars=10000)
```

**搜索特定关键词：**
```bash
export http_proxy="http://172.29.160.1:21882"
~/.openclaw/skills/freeUnlimited-websearch/search.py "keyword"
```

## 4. 记忆系统维护规范

### 三层记忆架构

```
┌─────────────────────────────────┐
│  MEMORY.md（长期记忆）            │
│  - 重要事件                      │
│  - 关键配置                      │
│  - 技术栈                        │
│  - 用户信息                      │
└─────────────────────────────────┘
          ↓ 定期更新
┌─────────────────────────────────┐
│  memory/YYYY-MM-DD.md（每日日志）│
│  - 当日工作                      │
│  - 待办事项                      │
│  - 问题解决                      │
└─────────────────────────────────┘
          ↓ 每日记录
┌─────────────────────────────────┐
│  USER.md（用户信息）              │
│  - 偏好设置                      │
│  - 环境信息                      │
│  - 习惯记录                      │
└─────────────────────────────────┘
```

### 更新时机

**MEMORY.md（长期记忆）：**
- 重要事件发生时
- 关键配置变更时
- 技术栈更新时
- 项目里程碑时

**每日日志（memory/YYYY-MM-DD.md）：**
- 每天结束时总结
- 记录当日工作
- 更新待办事项
- 标记问题解决

**USER.md（用户信息）：**
- 用户明确表达偏好时
- 环境配置变更时
- 用户说"记住这个"时

### 关键原则

1. **"下次别让我再提醒" → 立即写入 USER.md**
2. **解决了重要问题 → 写入 MEMORY.md**
3. **每日复盘 → 更新当日日志**
4. **沉淀知识点 → 写入知识库**

### 维护检查清单

- [ ] MEMORY.md 是否包含所有重要信息？
- [ ] 每日日志是否更新？
- [ ] USER.md 是否记录用户偏好？
- [ ] 知识库是否有新增内容？

## 5. 用户需求响应模式

### 典型场景

**用户说：** "这条信息保存到记忆中，下次别让我再提醒"

### 标准响应流程

```
1. 立即更新记忆系统
   ├─ USER.md（用户偏好）
   └─ MEMORY.md（重要信息）

2. 确认保存成功
   └─ 检查文件是否更新

3. 告知用户
   └─ "已记住，以后自动执行"
```

### 示例

**用户指令：** "我的桌面路径是 /mnt/d/桌面，记住这个"

**正确响应：**
```bash
# 1. 更新 USER.md
echo "- 桌面路径：/mnt/d/桌面/" >> USER.md

# 2. 更新 MEMORY.md
echo "- 桌面路径：/mnt/d/桌面/" >> MEMORY.md

# 3. 确认
ls /mnt/d/桌面/

# 4. 告知用户
echo "✓ 已记住桌面路径，以后直接使用"
```

**错误响应：**
1. ❌ 只记在脑子里（下次就忘了）
2. ❌ 只更新一个地方（不够全面）
3. ❌ 没有确认（用户不知道记没记）

### 模式总结

```
用户指令 + "记住/下次不用再说"
    ↓
立即写入记忆系统（USER.md + MEMORY.md）
    ↓
确认保存成功
    ↓
告知用户"已记住，以后自动执行"
```

### 常见用户偏好记录

- 桌面路径：`/mnt/d/桌面/`
- 时区：`GMT+8 (Asia/Shanghai)`
- 操作系统：`WSL2 (Ubuntu 22.04)`
- 代理配置：西部世界 VPN
- 常用工具：OpenClaw、飞书
